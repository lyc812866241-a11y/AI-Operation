# 系统性调试规范 (Systematic Debugging)

> **触发条件**：遇到任何 Bug、测试失败或非预期行为时，在提出修复方案之前，必须先执行本规范。
>
> **铁律**：`NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST`
> 未完成 Phase 1，不允许提出任何修复方案。症状修复等于失败。

---

## 何时使用

**以下情况必须使用本规范：**
- 测试失败
- 生产环境 Bug
- 非预期行为
- 性能问题
- 构建失败
- 模块集成问题

**尤其是以下高风险情况，更不能跳过：**
- 时间紧迫（紧急情况最容易诱发猜测性修复）
- "只需要改一个地方"看起来显而易见时
- 已经尝试了多次修复
- 上一个修复没有生效
- 自己没有完全理解问题所在

**不允许跳过的借口：**
- "问题很简单"（简单问题同样有根因）
- "时间不够"（系统性调试比反复试错更快）
- "用户要求立刻修复"（系统性方法比乱猜更快）

---

## 四阶段调试流程

**每个阶段必须完成后才能进入下一阶段。**

### Phase 1：根因调查（Root Cause Investigation）

**在尝试任何修复之前：**

1. **完整阅读错误信息**
   - 不要跳过错误或警告
   - 错误信息通常直接包含解决方案
   - 完整阅读 stack trace
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 能否稳定触发？
   - 精确的复现步骤是什么？
   - 是否每次都发生？
   - 无法稳定复现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - 什么变更可能导致了这个问题？
   - `git diff`、近期 commit 记录
   - 新依赖、配置变更、环境差异

4. **多组件系统：加入诊断埋点**

   当系统有多个组件（如：管线节点 A → B → C）时，**在提出修复方案之前**，必须先加入诊断日志：

   ```
   对每个组件边界：
     - 记录进入该组件的数据
     - 记录离开该组件的数据
     - 验证环境变量/配置的传递
     - 检查每一层的状态

   运行一次，收集证据，确定问题发生在哪一层
   然后分析证据，定位故障组件
   然后针对该具体组件深入调查
   ```

5. **追溯数据流**
   - 错误值从哪里产生？
   - 是谁用错误值调用了这里？
   - 沿调用链向上追溯，直到找到源头
   - 在源头修复，而不是在症状处修复

### Phase 2：模式分析（Pattern Analysis）

**找到模式，再动手修复：**

1. **寻找可工作的对照样本**
   - 在同一代码库中找到类似的、正常工作的代码
   - 什么相似的东西是正常的？

2. **对照参考实现**
   - 如果是在实现某个模式，完整阅读参考实现，不要略读
   - 完全理解模式后再应用

3. **识别差异**
   - 正常代码和故障代码之间有什么不同？
   - 列出每一处差异，无论多小
   - 不要假设"那个不可能有影响"

4. **理解依赖关系**
   - 这个组件依赖什么？
   - 需要什么配置、环境变量、前置条件？
   - 它做了哪些隐含假设？

### Phase 3：假设与验证（Hypothesis & Testing）

**用科学方法：**

1. **提出单一假设**
   - 明确陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体，不要模糊

2. **最小化验证**
   - 做最小的改动来验证假设
   - 每次只改一个变量
   - 不要同时修复多个问题

3. **验证后再继续**
   - 有效？→ 进入 Phase 4
   - 无效？→ 提出新假设
   - 不要在原有修复上叠加更多修复

4. **不确定时**
   - 直接说"我不理解 X"
   - 不要假装知道
   - 请求帮助或进一步调研

### Phase 4：实施修复（Implementation）

**修复根因，而不是症状：**

1. **先创建失败测试**
   - 最简单的可复现用例
   - 尽量写成自动化测试
   - 修复前必须有这个测试
   - 参考 `skills/test-driven-development/SKILL.md`

2. **实施单一修复**
   - 针对已识别的根因
   - 每次只改一处
   - 不做"顺手"的改进
   - 不捆绑重构

3. **验证修复**
   - 测试通过了吗？
   - 其他测试没有被破坏吗？
   - 问题真的解决了吗？

4. **修复无效时**
   - **停止**
   - 统计：已经尝试了几次修复？
   - 少于 3 次：带着新信息回到 Phase 1
   - **3 次或以上：停止，质疑架构（见下方第 5 条）**
   - 不允许在没有架构讨论的情况下尝试第 4 次修复

5. **3 次修复均失败：质疑架构**

   **这种模式表明存在架构问题：**
   - 每次修复都暴露出不同位置的新问题（共享状态/耦合）
   - 修复需要"大规模重构"才能实施
   - 每次修复都在其他地方产生新症状

   **停止，质疑根本：**
   - 这个架构模式本身是否合理？
   - 我们是否只是因为惯性而坚持它？
   - 应该重构架构，而不是继续修复症状？

   **在尝试更多修复之前，与用户讨论。**

## 红色警报：出现以下想法时立刻停止

- "先快速修一下，之后再调查"
- "试着改 X 看看有没有效果"
- "加几个改动，跑一下测试"
- "跳过测试，我手动验证一下"
- "可能是 X，让我修一下"
- "我没完全理解，但这个可能有用"
- "参考说 X，但我稍微调整一下"
- "主要问题是这些：[列出修复方案，但没有调查]"
- 在追溯数据流之前就提出解决方案
- **"再试一次修复"（已经试了 2 次以上时）**
- **每次修复都在不同地方暴露新问题**

**以上所有情况：停止，回到 Phase 1。**

## 常见借口对照表

| 借口 | 现实 |
|---|---|
| "问题很简单，不需要流程" | 简单问题同样有根因。流程对简单 Bug 反而更快。 |
| "紧急情况，没时间走流程" | 系统性调试比猜测-试错循环更快。 |
| "先试试这个，再调查" | 第一次修复就定下了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复不会持久。先写测试才能证明修复有效。 |
| "同时做多个修复节省时间" | 无法隔离是哪个有效。会引入新 Bug。 |
| "参考太长了，我来调整模式" | 部分理解必然导致 Bug。完整阅读。 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因。 |
| "再试一次"（已失败 2 次以上） | 3 次失败 = 架构问题。质疑模式，不要再修了。 |

## 快速参考

| 阶段 | 核心活动 | 完成标准 |
|---|---|---|
| **1. 根因调查** | 读错误、复现、检查变更、收集证据 | 理解了"是什么"和"为什么" |
| **2. 模式分析** | 找对照样本、对比差异 | 识别了差异点 |
| **3. 假设验证** | 提出假设、最小化测试 | 假设得到确认或被推翻 |
| **4. 实施修复** | 创建测试、修复、验证 | Bug 解决，测试通过 |
