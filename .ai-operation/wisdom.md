> **二阶 / scope = 所有项目** — 跨项目通用方法论、原则、智慧。
>
> **铁律(议题 #009)**:
> 1. **type 边界 = scope 边界**——单项目 scope 的内容(corrections)无论累积多少次,都不会自动升级到这里

## 1. 求导思维(议题 #005)

遇到问题时,**按四阶向高处求导**:

| 阶 | 内容 | scope |
|---|---|---|
| 一阶 | 解决具体问题 | 单项目 |
| 二阶 | 解决产生原因(根因) | 单项目 |
| 三阶 | 沉淀为机制(项目内抽象规则) | 单项目 → corrections §1 |
| 四阶 | 沉淀到跨项目智慧(普适方法论) | **所有项目** → wisdom.md |

**三阶 → 四阶之间是 scope 跃迁,不是简单抽象升级**(议题 #005 v2 修正)。
跃迁时必须回答:**"这个机制对其他项目也成立吗?"** 经得起拷问才能进 wisdom。

### 三个纪律

#### 纪律 1:成本指数上升,顶尖能力是判断"爬到几阶"

| 问题性质 | 该爬到几阶 |
|---|---|
| 一次性,不会再遇到 | 一阶就停 |
| 偶尔出现 | 爬到二阶,记笔记 |
| 反复出现 | 必须三阶,提炼项目内机制 |
| 多项目 / 多场景共有 | 四阶,固化进 wisdom |

**判断"该爬到几阶" = taste 应用**。对每个 bug 都做四阶 = 瘫痪在元工作里。

#### 纪律 2:爬之前必须"三角定位"真原因

二阶之前,先看 **3 个以上类似问题**,看共性是什么。否则你不是在求导,是在**自我说服**。

#### 纪律 3:爬到顶后必须"反向验证"

四阶沉淀进 wisdom 后,**必须用 1-2 个具体场景验证它真的工作**,否则是空中楼阁。

### 完整闭环

```
   四阶: wisdom.md      ←──────┐
       ↑                       │ 反向验证
       │ scope 跃迁(type 转换) │
   ─────────────────────────── │  scope 边界
       ↑                       │
   三阶: corrections §1       │
       ↑                       │
       │ 抽象                   │
   二阶: 根因分析             │
       ↑                       │
       │ 抽象                   │
   一阶: 具体问题  ──────────┘
```

## 2. 沟通用自然语言,代码留给落档

**核心命题**:向用户描述问题、方案、设计选择时,**必须用自然语言**;**禁止用代码语言**(代码块、伪代码、API 签名堆叠)代替沟通。

### 推理

- **自然语言被人脑消化更快**——议题 #002 思维链连续性的人侧:用户的认知带宽是稀缺资源,代码语言把"翻译代码 → 理解意图"的成本嫁接给用户
- **代码语言适合"实施",不适合"沟通"**——真正要执行的代码,会落到文件里被工具读取;对话讨论"做什么"应该用人能直接判断/反驳的形式
- **议题 #004 的延伸**——人脑做判断/决策,框架做执行;沟通是判断层的活,不是执行层的活;代码语言污染判断过程
- **议题 #005 求导思维**:讨论二/三阶时,代码是细节。在更高阶用代码语言 = 把读者拽回一阶,**反求导**

### 应用范围

- 描述问题(发生了什么、影响是什么)
- 提议方案(选项 A vs B 的优劣、tradeoff)
- 解释设计选择(为什么这么做)
- 状态报告 / 进度更新 / 诊断结论
- 元工作的所有沟通

### 反例边界(以下情况代码语言**对的**)

- 真要落到文件里被执行的代码改动(那是实施,不是沟通)
- 用户具体问"这个语法/API 怎么写" → 代码示例必要
- Debug 引用 traceback / 代码片段做证据
- 用户主动要求看代码

### 反模式(被禁止)

- 用代码块伪装解释(`if condition: do_X` 代替"如果 X 就做 Y")
- 用类型签名堆叠代替描述(`Foo<Bar<Baz>>` 代替"嵌套结构")
- 用 ASCII art / 示意代码块代替自然语言描述(除非它真的传递了图表无法替代的信息)
